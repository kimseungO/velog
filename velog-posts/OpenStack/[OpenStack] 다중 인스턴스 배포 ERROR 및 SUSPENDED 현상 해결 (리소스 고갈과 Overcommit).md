### 문제

OpenStack 환경에 Kubernetes 구축을 위해 4GB 메모리(`k8s-flavor`)를 가진 인스턴스를 순차적으로 배포하던 중 문제가 발생했습니다. 
마스터 노드와 첫 번째 워커 노드(`k8s-worker-01`)까지는 `ACTIVE` 상태로 잘 올라왔으나, **두 번째 워커 노드(`k8s-worker-02`)를 배포하자마자 즉시 `ERROR` 상태**에 빠졌습니다.

심지어 마스터 노드마저 갑자기 `SUSPENDED`(일시 정지) 상태로 변했습니다.

```bash
$ openstack server list
+--------------------------------------+-----------------+-----------+---------------------------+--------------+------------+
| ID                                   | Name            | Status    | Networks                  | Image        | Flavor     |
+--------------------------------------+-----------------+-----------+---------------------------+--------------+------------+
| 9f0daf8b-660c-4074-964f-d0faf5051174 | k8s-worker-02   | ERROR     |                           | Ubuntu-22.04 | k8s-flavor |
| 9ab13b20-535d-439e-a5cf-83c1de05f3da | k8s-worker-01   | ACTIVE    | public_net=192.168.95.138 | Ubuntu-22.04 | k8s-flavor |
| b5934935-c4c1-41f1-8f2c-5cd805a6b3a9 | k8s-master-node | SUSPENDED | public_net=192.168.95.111 | Ubuntu-22.04 | k8s-flavor |
+--------------------------------------+-----------------+-----------+---------------------------+--------------+------------+
```

<br>

### 원인

가상화 환경에서 자원을 빡빡하게 할당했을 때 흔히 발생하는 **물리적/논리적 자원 고갈(Out of Memory)** 현상으로 추측할 수 있습니다.

정확한 원인을 파악하기 위해 OpenStack의 Placement 서비스에 컴퓨트 노드(Compute01, Compute02)의 리소스 인벤토리를 질의해 보았습니다.

```bash
$ openstack resource provider inventory list <Compute01_UUID>
+----------------+------------------+----------+----------+----------+-----------+-------+------+
| resource_class | allocation_ratio | min_unit | max_unit | reserved | step_size | total | used |
+----------------+------------------+----------+----------+----------+-----------+-------+------+
| MEMORY_MB      |              1.0 |        1 |     7901 |      512 |         1 |  7901 | 4096 |
| DISK_GB        |              1.0 |        1 |       37 |        0 |         1 |    37 |   20 |
+----------------+------------------+----------+----------+----------+-----------+-------+------+
```

**1. `k8s-worker-02`의 ERROR 원인 (NoValidHost)**

  * 각 컴퓨트 노드의 총 메모리는 약 8GB(`7901 MB`)였습니다.
  * 여기에 OS 예약 용량(`512 MB`)과 이미 올라가 있는 마스터/워커1 노드의 사용량(`4096 MB`)을 빼면 **남은 가용 램은 약 3.3GB**에 불과했습니다.
  * 세 번째 인스턴스가 요구하는 램은 4GB였으므로, Nova 스케줄러가 인스턴스를 넣을 공간을 찾지 못하고 에러를 발생시켰습니다.

**2. 마스터 노드의 SUSPENDED 원인**

  * 가상 환경(VMware) 위에서 OpenStack을 구동할 때, 호스트의 물리 메모리가 한계치까지 차오르면 하이퍼바이저가 호스트 OS의 다운을 막기 위해 실행 중인 VM 중 하나를 강제로 `SUSPEND`(일시 정지) 시켜버리는 보호 로직이 작동한 결과입니다.

<br>

### 해결

문제를 해결하고 향후 쿠버네티스 파드(Pod) 배포 시의 안정성을 확보하기 위해, '물리적 스펙업'과 클라우드의 논리적 기술인 '오버커밋(Overcommit)'을 동시에 적용했습니다.

**Step 1. 하이퍼바이저(VMware) 물리 리소스 증설**

  * 가장 확실한 방법으로, 컴퓨트 노드들의 전원을 끄고 VMware 설정에서 RAM을 12GB, 디스크를 60GB로 넉넉하게 증설해 주었습니다.

**Step 2. OpenStack 논리적 오버커밋(Overcommit) 설정 (Kolla-Ansible)**

  * 물리 자원 이상으로 가상머신을 밀어 넣을 수 있도록 초과 할당 비율(`allocation_ratio`)을 늘려줍니다. Controller 노드에서 설정을 오버라이드합니다.
    ```bash
    $ sudo mkdir -p /etc/kolla/config
    $ sudo vi /etc/kolla/config/nova.conf
    ```
    ```ini
    [DEFAULT]
    ram_allocation_ratio = 1.5
    disk_allocation_ratio = 1.5
    ```
  * 설정 파일 저장 후 Kolla-Ansible 재구성을 통해 전체 노드에 반영합니다.
    ```bash
    $ kolla-ansible -i multinode reconfigure
    ```

**Step 3. 리소스 확인 및 재배포**

  * 재구성 후 다시 인벤토리를 확인해 보면 `allocation_ratio`가 1.5로 늘어났고, 전체 램(`12047 MB`) 여유 공간이 대폭 확장된 것을 확인할 수 있습니다.

    ```bash
    $ openstack resource provider inventory list <Compute01_UUID>
    +----------------+------------------+----------+----------+----------+-----------+-------+------+
    | resource_class | allocation_ratio | min_unit | max_unit | reserved | step_size | total | used |
    +----------------+------------------+----------+----------+----------+-----------+-------+------+
    | MEMORY_MB      |              1.5 |        1 |    12047 |      512 |         1 | 12047 | 4096 |
    | DISK_GB        |              1.5 |        1 |       37 |        0 |         1 |    37 |   20 |
    +----------------+------------------+----------+----------+----------+-----------+-------+------+
    ```

  * 에러 상태였던 VM을 지우고 다시 생성하면, 세 대의 인스턴스가 모두 여유롭게 배치되어 **ACTIVE** 상태가 된 것을 볼 수 있습니다.

    ```bash
    $ openstack server list
    +--------------------------------------+-----------------+--------+---------------------------+--------------+------------+
    | ID                                   | Name            | Status | Networks                  | Image        | Flavor     |
    +--------------------------------------+-----------------+--------+---------------------------+--------------+------------+
    | cd9d8caa-dcc0-48dc-8391-722b1a9ff163 | k8s-worker-02   | ACTIVE | public_net=192.168.95.169 | Ubuntu-22.04 | k8s-flavor |
    | 9ab13b20-535d-439e-a5cf-83c1de05f3da | k8s-worker-01   | ACTIVE | public_net=192.168.95.138 | Ubuntu-22.04 | k8s-flavor |
    | b5934935-c4c1-41f1-8f2c-5cd805a6b3a9 | k8s-master-node | ACTIVE | public_net=192.168.95.111 | Ubuntu-22.04 | k8s-flavor |
    +--------------------------------------+-----------------+--------+---------------------------+--------------+------------+
    ```

성공적으로 리소스 병목을 해결하고 클러스터 기반 인프라 배포를 완료했습니다.
