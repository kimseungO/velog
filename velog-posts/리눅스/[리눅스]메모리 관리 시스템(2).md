## 가상 메모리
### 가상 메모리가 없을 때 생기는 문제점
- 메모리 단편화
- 멀티 프로세스 구현이 어려움
- 비정상적인 메모리 접근

#### 메모리 단편화
![](https://velog.velcdn.com/images/rtd7878/post/b07b44f7-5109-4949-aa5b-635b5945c45a/image.png)

- 그림에서 비어 있는 메모리는 합치면 300바이트이지만 각자 따로따로 100바이트씩 3개의 영역으로 나눠진 상태이므로 100바이트보다 큰 영역을 화곱하려면 실패한다.
- 비어 있는 영역 3개를 하나로 묶어서 다루면 다음과 같은 이유로 불가능 하다.
  - 프로그램이 메모리를 확보할 때마다 확보한 메모리가 몇 개의 영역으로 쪼개져 있는지 일일이 관리해야 하므로 불편함.
  - 크기가 100바이트보다 큰 연속된 데이터 묶음, 예를 들어 300바이트짜리 배열을 작성하려는 용도로 사용 불가능.
  
#### 멀티 프로세스 구현이 어려움
![](https://velog.velcdn.com/images/rtd7878/post/1c078669-82cf-44f3-b3c1-d51870cb39c9/image.png)
- 그림처럼 프로세스 A를 실행했을 때 코드 영역이 주소 300~400이고, 데이터가 주소 300~500으로 매핑된 상태일 때,
이 다음에 동일한 실행 파일을 사용해서 별도의 프로세스 B를 실행했다고 가정하면, 이는 불가능하다.
- 그 이유는 이 프로그램은 주소 300~500에 매핑된다고 전제하는데, 이미 해당 영역은 프로세스 A가 사용하고 있기 때문.
- 결국 동시에 여러 프로그램을 실행하려면, 사용자가 모든 프로그램의 배치 장소가 겹치지 않도록 의식해서 관리해야 한다는 말이 된다.

#### 비정상적인 메모리 접근
![](https://velog.velcdn.com/images/rtd7878/post/a2795bfd-dfdd-4533-8050-e372ff9db886/image.png)


- 커널이나 수많은 프로세스가 메모리에 존재할 때, **자신이 사용하는 영역이 아님에도 불구하고 접근할 수 있는 문제**가 생김.

### 가상 메모리 기능
![](https://velog.velcdn.com/images/rtd7878/post/88195876-ecd7-42eb-ac55-830a10d92dde/image.png)

- 가상 메모리는 프로세스가 메모리에 접근할 때 시스템에 설치된 메모리에 직접 접근하는 대신에 **가상 주소**를 사용해서 간접적으로 접근하는 기능.
- 가상 주소에 대비되는 시스템에 설치된 메모리의 실제 주소를 **물리주소**라고 하며, 이런 주소를 사용해서 접근 가능한 범위를 **주소공간** 이라고 함.
- 그림에서프로세스가 주소 100에 접근하면, 실제로 물리 메모리에서는 주소600에 존재하는 데이터에 접근.
- 프로세스에서 실제 물리 메모리에 직접 접근하는 방법은 없다.

#### 페이지 테이블
![](https://velog.velcdn.com/images/rtd7878/post/07a38c37-fac2-487b-aed0-ce7b7adc3c06/image.png)
- 가상 주소를 물리 주소로 변환하려면 커널 메모리 내부에 저장된 페이지 테이블을 사용.
- 페이지 테이블에서 한 페이지에 대응하는 데이터를 **페이지 테이블 엔트리**라고 부름.
  - 페이지 테이블 엔트리는 **가상 주소**와 **물리 주소** 대응 정보를 포함.
- 페이지 크기는 CPU 아키텍처에 따라 정해져 있음.
  - x89_64 아키텍처는 4KiB
  - 여기선 설명이 편하도록 페이지 크기를 100바이트로 정함.
- 페이지 테이블은 커널이 작성.

![](https://velog.velcdn.com/images/rtd7878/post/35fda789-05e4-4f1d-9f0e-e7af4c0c316c/image.png)

- 주소 300~500에 프로세스가 접근하면 CPU에서 **페이지 폴트**라고 하는 예외가 발생.
- 이런 페이지 폴트 예외가 발생하면 CPU에서 실행 중인 명령이 중단되고, 커널 메모리에 배치된 **페이지 폴트 핸들러** 처리가 실행 됨.
![](https://velog.velcdn.com/images/rtd7878/post/1e38cfbd-dff0-421d-be4a-98ca57946af9/image.png)

- 커널은 페이지 폴트 핸들러를 이용해서 프로세스가 비정상적인 메모리 접근을 일으켰다는 걸 감지.
- 그런 이후에 SIGSEGV 시그널을 프로세스에 송신.
- SIGSEGV 시그널을 받은 프로세스는 보통은 강제 종료됨.

### 가상 메모리로 문제 해결하기

#### 메모리 단편화
- 프로세스의 페리지 테이블을 잘 설정하면 물리 메모리의 단편화된 영역이라도 프로세스 가상 주소 공간에서는 커다란 하나의 영역으로 다룰 수 있다. => 단편화 문제 해결

#### 멀티 프로세스 구현이 어려움
- 가상 주소 공간은 프로세스마다 만들어진다.
- 따라서 멀티 프로세스 환경에서 각자의 프로그램이 다른 프로그램과 주소가 중복되는 걸 피할 수 있다.

#### 비정상적인 메모리 접근
- 프로세스마다 가상 주소 공간이 있다는 말은 애초에 다른 프로세스의 메모리가 어떻게 되어 있는지 알 수가 없기 때문에 접근할 수 없다는 뜻이다.
- 따라서 어떤 프로세스에서 다른 프로세스의 메모리로 비정상적인 접근이 불가능하다.
