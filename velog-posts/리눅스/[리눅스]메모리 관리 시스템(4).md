## 페이지 테이블 계층화

- x86_64 아키텍처라면 가살 주소 공간 크기는 128Tib까지, 1페이지 크기는 4KiB, 페이지 테이블 엔트리 크기는 8바이트.
- 따라서 단순히 계산하면 프로세스 1개당 페이지 테이블에 256GiB라는 메모리가 필요하다는 계산이 됨.
=> 계층형 페이지 테이블을 사용하여 해소. (x86_64 아키넥처는 4단 구조 페이지 테이블 사용)
![](https://velog.velcdn.com/images/rtd7878/post/b21f539f-48c3-4acd-baaf-369010e4762d/image.png)

### Huge Page
- 앞서 말했듯 프로세스가 확보한 메모리 용량이 늘어나면 해당 프로세스의 페이지 테이블이 사용하는 물리 메모리 사용향도 늘어남.
- 이런 문제를 해결하기 위해 리눅스는 Huge Page 구조를 이용
- 위 구조를 다 채우면 테이블 계층 1단은 건너 뛰고 다음과 같은 테이블이 됨.
![](https://velog.velcdn.com/images/rtd7878/post/f3e5c22f-73d4-4233-b7a8-3187c7f1b872/image.png)
- 페이지 테이블 엔트리 개수가 20개에서 4개로 줄어들음.
- 메모리 사용량 하락
- fork() 함수에서 페이지 테이블을 복사하는 비용도 줄어서 fork()함수 속도 향상

### Transparent Huge Page(THP)
- Huge Page를 사용하려면 메모리 확보시 일일이 요청해야하므로 번거로움
- THP 기능은 가상 주소 공간 내부에 있는 연속된 여러 4KiB 페이지가 어떤 조건을 만족하면 그걸 하나로 묶어서 자동으로 Huge Page로 바꿔줌.
- 여러 페이지를 합쳐서 하나의 Huge Page로 만드는 데 들어가는 처리 비용, 그리고 Huge Page를 구성하던 조건이 성립하지 않게 되면 커다란 페이지를 4GiB 페이지로 다시 분할하는 처리 때문에 **부분적으로 성능이 떨어지는 경우가 있음**
=> 따라서 시스템 관리자가 선택적으로 사용 여부 결정
