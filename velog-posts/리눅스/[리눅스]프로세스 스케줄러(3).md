## 컨텍스트 스위치
> 논리 CPU에서 동작하는 프로세스가 전환되는 것을 컨텍스트 스위치라고 합니다.

## 처리 성능
> - 턴어라운드 타임(turnaround time): 처리(반환) 시간, 시스템에 처리를 요청했을 때부터 처리가 끝날 때까지 걸린 시간
- 스루풋(throughput): 처리량. 단위 시간당 처리를 끝낸 개수 
(프로세스 개수/[sec/min...{경과 시간}])


처리 성능을 직접 측정하기 위한 실습.
실습에는 multiload.sh 프로그램을 대상으로 다음과 같은 성능 정보를 수집합니다.
- 평균 턴어라운드 타임: 모든 부하 처리의 real(경과 시간) 값의 평균
- 스루풋: 프로세스 개수를 multiload.sh(프로세스 실행에 걸린 시간 출력) 프로그램의 real 값으로 나눈 값 --> 프로세스 개수/real(경과 시간)

### 실습 프로그램
<span style="font-size:12px">실습 소스: https://www.hanbit.co.kr/support/supplement_survey.html?pcode=B9151150768</span>
cpuperf.sh 프로그램 사용법: cpuperf.sh [-m] <최대 프로세스 개수>

1. cpuperf.data 파일에 성능 정보를 저장.
    - 엔트리 개수는 <최대 프로세스 개수>
    - 출력 형식은 <프로세스 개수> <평균 턴어라운드 타임[초]> <스루풋[프로세스/초]>
2. 성능 정보를 바탕으로 평균 턴어라운드 타임 그래프를 만들어서 avg-tat.jpg에 저장.
3. 성능 정보를 바탕으로 스루풋 그래프를 만들어서 throughput.jpg에 저장.
4. -m 옵션은 multiload.sh 프로그램에 그대로 전달.

#### 실습1
논리 CPU 1개만 사용하고 최대 프로세스 개수를 8로 지정해서 부하 처리 프로세스를 실행
```bash
$ ./cpuperf.sh 8
```
![](https://velog.velcdn.com/images/rtd7878/post/5149c7d4-df86-4aae-9fba-1581e09607e1/image.png)
- 논리 CPU 1개, 최대 프로세스 개수가 8일 때 평균 턴어라운드 타임(총 처리 시간의 평균)

![](https://velog.velcdn.com/images/rtd7878/post/dc4d8248-60c5-45f2-b9c4-453b5c9d4515/image.png)

- 논리 CPU 1개, 최대 프로세스 개수가 8일 때 스루풋(프로세스 개수/초)

#### 분석
- 결과를 살펴보면 논리 CPU 개수보다 프로세스 개수가 많아지면 평균 처리 시간만 늘어나고 스루풋은 차이가 없음을 알 수 있다.
- 이후 프로세스 개수를 더 늘리면 스케줄러가 발생시킨 컨텍스트 스위치 때문에 평균 처리 시간이 점점 길어지고 스루풋도 계속 떨어진다.
=>성능 관점에서 CPU 자원을 모두 사용시 프로세스 개수를 늘린다고 성능 문제를 해결할 수 없다는 뜻.

--- 
#### 실습2
아래 명령어를 통해 논리 CPU 개수 확인
```bash
$ grep -c processor /proc/cpuinfo
8
```

최대 프로세스 개수: 8, 모든 CPU: 8 를 사용해서 성능 정보를 수집
```bash
$ ./cpuperf.sh -m 8
```
![](https://velog.velcdn.com/images/rtd7878/post/a557097d-e5b4-4205-8419-4b6675e38ce8/image.png)


- 모든 논리 CPU, 최대 프로세스 개수가 8일 때 평균 턴어라운드 타임

![](https://velog.velcdn.com/images/rtd7878/post/b953758d-aa11-4aae-a046-165868996200/image.png)

- 모든 논리 CPU, 최대 프로세스 개수가 8일 때 스루풋

#### 분석
- 턴어라운드 타임 그림을 보면 프로세스 개수가 논리 CPU 개수(여기서는 4)와 같아질 때까지는 평균 처리 시간이 천천히 늘어나지만 그 후는 갑자기 길어지는 걸 알 수 있다.
- 스루풋은 프로세스 개수와 논리 cpu 개수가 같아질 때 까지 향상되지만 그 후에는 한 풀 꺾이는 걸 알 수 있다.

=> 논리 CPU가 많이 내장된 경우라도 충분한 수의 프로세스가 실행되어야 비로소 스루풋이 향상된다.
=> 무조건 프로세스 개수를 늘린다고 스루풋은 개선되지 않는다.

### 결과
#### 프로그램 병렬 실행의 중요성
예전에는 새로운 CPU가 나올 때마다 논리 CPU당 성능이 극적으로 향상되었다. 하지만 최근에는 성능 향상이 어려워졌고, 따라서 CPU 코어 개수를 늘리는 방법 등으로 CPU 전체 성능을 끌어올리는 쪽으로 방향이 바뀌었다.
커널도 이런 시대 흐름에 발맞춰서 코어 개수가 늘어난 경우의 확장성을 향상시켜 왔다.
